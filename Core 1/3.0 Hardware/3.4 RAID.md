## Data redundancy
- **Hard drives store huge amounts of data**
	- Important data
- **Hard drives are moving components**
	- They will eventually break
- **What happens to the data when the drive fails?**
	- Requires preparation
	- Use an array of drives
- <mark>RAID is not a backup</mark>
	- Need to maintain a completely separate backup process

## RAID
- <mark>Redundant Array of Independent Disks</mark>
	- They're also inexpensive disks
- Storage is another very important part of fault tolerance
- Data loss prevention
	- If a disk failure happens, no data will be lost.
- **Different RAID levels**
	- Some redundant, some are not
- <mark>RAID 0</mark> – Striping
- <mark>RAID 1 </mark>– Mirroring
- <mark>RAID 5</mark> - Striping with one parity drive
- <mark>RAID 6</mark> - Striping with two parity drives
- <mark>RAID 10 (1+0)</mark> – Nested RAID / A stripe of mirrors

### RAID 0 - Striping
![588](<./assets/3.4 RAID/file-20260312170720403.png>)[^1]
- <mark>File blocks (data) are split between two or more physical drives</mark>
- ***Not fault-tolerant***
- ***Data is 'striped' across multiple disks***
- **High performance**
	- Data written quickly
	- Accessing data is much faster
	- The advantage is *speed*
- **No redundancy**
	- ![](<./assets/3.4 RAID/file-20260312171228367.png>)[^1]
	- A drive failure breaks the array
	- RAID 0 is zero redundancy

### RAID 1 - Mirroring
![553](<./assets/3.4 RAID/file-20260312171500892.png>)[^1]
- <mark>File blocks (data) are duplicated between two or more physical drives</mark>
- ***Fault tolerant***
- *Need at least 2 drives*
- **High disk utilization**
	- Every file is duplicated
	- Required disk space is doubled
- **High redundancy**
	- ![](<./assets/3.4 RAID/file-20260312171551808.png>)[^1]
	- Drive failure does not affect data availability


### RAID 5 - Striping with parity
![540](<./assets/3.4 RAID/file-20260312172107527.png>)[^1]
- <mark>File blocks (data) are striped across multiple disks</mark>
	- Along with a ***parity*** block
		- It is used to rebuild the data in the event of a disk failure
	- *Requires at least 3 drives*
- ***The equivalent of an entire disk is used to store parity.***
	![](<./assets/3.4 RAID/file-20260312174201856.png>)[^2]
	- <u>Example</u>: In a RAID 5 array of four physical drives, three of them would have the actual data, while the fourth one would carry parity data.
	- <u>Example</u>: An array of 4 disks totaling 4 terabytes, only 3 terabytes will be used for actual data storage.
	- <u>Example</u>: With an array of 3 disks, the equivalent size of 2 of the disks is actually used to store data. The equivalent size of one of the disks is used to store parity.
- **Efficient use of disk space**
	- The parity drives are distributed across the multiple drives
	- Files aren't duplicated, but space is still used for parity
	- No need to duplicate all of the data
- **High redundancy**
	- Data is available after a drive failure
		- Use existing data and combine it with parity
			![](<./assets/3.4 RAID/file-20260312173316714.png>)[^2]
			- Be able to recreate the data that's no longer there
		- Unfortunately cannot handle more than 1 disk failure
			![](<./assets/3.4 RAID/file-20260312173458628.png>)[^2]
	- **Parity calculation may affect performance** 
		- Requires a bit of CPU overhead
		- There may be times when using RAID 5 in recovery mode results in a performance hit.

### RAID 6 - Striping with double parity
![](<./assets/3.4 RAID/file-20260312174722694.png>)[^2]
- **Similar to RAID 5**
- <mark>Adds another parity block</mark>
	- Parity is spread twice on all the disks
	- ***Requires at least 4 drives***
- ***The equivalent size of 2 disks is actually used to store data.***
	![](<./assets/3.4 RAID/file-20260312175415937.png>)[^2]
	- The equivalent size of 2 disks are used to store the double parity
	- <u>Example</u>: If 4 disks are 4TB each, only 2TB would be used to store data and the other 2TB would be used double parity 
- **Could lose two physical drives**
	![](<./assets/3.4 RAID/file-20260312174836238.png>)[^2]
	- Data would continue to be available
	- Replace the failed disks, then RAID 6 would use the double parity from the other disks to rebuild the data on the new drives.
- **Requires another drive**
	- Adds more parity
	- Unfortunately does not add more capacity
		- The read performance from RAID 5 & 6 are about the same.
		- As for writing data, RAID 6 suffers greatly
			- RAID 6 writes data slower than RAID 5 since it has to write 2 independent parity blocks instead of 1.

### RAID 10 (RAID 1+ 0)
![](<./assets/3.4 RAID/file-20260312172731345.png>)[^1]
- **Combines [RAID 0](<### RAID 0 - Striping>) with [RAID 1](<### RAID 1 - Mirroring>)**
	- Benefits from the fault tolerance of *RAID 1* and the speed of *RAID 0*
- <mark>The speed of striping, the redundancy of mirroring</mark>
	- A stripe of mirrors–the best of both worlds
- RAID 0 splits the file into different blocks
- RAID 1 will help mirror each striped set of drives
	- Resulting in separate individual drives that have a copy of the striped information
- ***Need at least 4 drives***
	- Can potentially lose multiple drives but still be up and running
- **You can only use 50% of the capacity for data storage**
	- <u>Example</u>: If you are using 4 disks in a RAID 10 setup, you can only use two of them for actual data storage.
